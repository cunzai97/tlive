# TLive：把 Claude Code / Codex / Pi 接进飞书话题的远程 AI 开发工作台

各位佬友大家好，最近我在折腾一个开源项目：**TLive**。

项目地址：<https://github.com/huanghuoguoguo/tlive>

**TLive 是一个飞书 / Lark 原生的远程 AI 开发工作台，用一个 bot 管理本机或远端节点上的 Claude Code、Codex 和 Pi。**

我做这个项目想解决的点很简单：别让手机硬模拟终端，把 AI coding agent 接进飞书 / Lark 本来就擅长的交互里。工作台用来选节点和会话，话题承载任务过程，按钮处理权限审批，群聊可以一起看和一起补需求。

## 为什么做 TLive

Claude Code、Codex 这类 agent 已经很好用了，但它们的主交互还是终端。人在电脑前还好，可一旦换到手机或团队协作场景，就会遇到一些很实际的问题：

- 手机 SSH 看输出、切目录、审批权限都不舒服。
- agent 的执行过程很难自然进入团队沟通上下文。
- 多台机器、多套工作区、多种 agent runtime 缺少统一入口。
- 会话恢复、历史查看、文件和图片交互，用纯命令行都不太自然。

TLive 的思路是：**飞书主窗口作为工作台，每个 AI 开发任务作为一个飞书话题。**

你在工作台里选择执行节点、选择 Claude Code / Codex / Pi、浏览目录、恢复历史；真正的任务进入独立话题里持续对话。一个话题就是一个 agent session。

## 为什么只做飞书 / Lark

TLive 最早是基于一个已有项目 fork 出来的，但现在代码结构和产品方向已经做了比较大的调整。

原项目里保留了多 IM 的思路，我这边后来基本把主线收敛到飞书 / Lark。这个取舍是有意为之：

- 维护多个 IM 适配器成本很高。
- 不同 IM 的卡片、按钮、话题、回调能力差异很大。
- TLive 的主要体验依赖飞书话题作为 session 容器。
- 目前我还没看到其他 IM 有同样好用的“话题 + 卡片 + 按钮 + 长连接 + 群聊”组合。

所以我现在把 TLive 收得很窄：先把飞书 / Lark 这条体验做好，定位就是 AI 远程开发工具。

## 我觉得最有价值的点

### 1. 尽量对齐移动端 AI 编程该有的操作习惯

TLive 的入口是飞书卡片工作台。

很多开源项目现在也在做类似的移动端入口，按钮、卡片、审批这些都不算稀奇。TLive 这里没有设计更复杂的一套交互，主要是把 Claude Code / Codex / Pi 在移动端常见的操作路径接到飞书里：

- 选择执行节点
- 新建 Claude Code / Codex / Pi 会话
- 查看当前目录
- 浏览目录并点击切换
- 查看最近会话
- 回到已有话题
- 恢复历史会话
- 处理权限审批
- 收发文件和图片
- 重启或升级远端 client

我更关心的是少折腾：选节点、开会话、审批、切目录、回历史，这些动作在手机上应该能自然完成，不需要退回到 SSH 里硬操作。

### 2. 飞书话题很适合承载 agent session

这个是我觉得 TLive 和很多远控工具最不一样的地方。

每次新建任务，TLive 会创建一个飞书话题。后续直接在这个话题里继续发消息。

实际用下来，这点很顺手：

- 一个话题就是一个 agent session。
- 不同任务不会混在一起。
- 任务过程、结果、权限审批都留在同一个上下文里。
- 可以 Pin 住重要话题。
- 旧会话可以从工作台回到话题。
- server 重启后，Pin 住的话题也可以作为恢复入口。

这比在一个聊天窗口里堆满所有任务要清楚很多。

### 3. 可以在群聊里共享 agent

我之前主要在单聊里用，最近试了群聊，发现这个方向其实很有意思。

放到项目群里，大概就是这个流程：

1. 群里 `@bot` 打开工作台。
2. 选择某台执行节点。
3. 新建 Claude Code / Codex / Pi 会话。
4. 任务进入话题。
5. 群成员都能看到执行过程和结果。
6. 后续可以在话题内继续补充需求。

相比远程展示一个终端，它的体验更接近在群里放了一个可以协作的 AI 开发成员。

权限和安全要自己把握，别把能操作生产环境的 agent 随便拉进大群。

### 4. Server / Client 架构：一个 bot 管多个执行节点

TLive 现在拆成了 server 和 client：

- **server** 负责飞书、工作台、话题、权限和控制面。
- **client** 负责真正执行 Claude Code、Codex 或 Pi。

client 可以跑在不同机器上，比如：

- 本地开发机
- 云主机
- 公司内网机器
- 不同项目的独立工作区

这些 client 都可以连到同一个 server，然后在飞书工作台里选择执行节点。

对我来说，这个能力很实用：一个飞书 bot 就能管理多个节点上的 agent，不需要每台机器都单独搞一套入口。

### 5. Claude Code / Codex / Pi 统一入口

TLive 目前支持：

- Claude Code
- Codex
- Pi

工作台会根据 client 上报的 provider 自动展示按钮。你不用在飞书里记很多命令，直接点对应按钮开会话。

不同 provider 能暴露的能力不完全一样，TLive 会尽量按 provider 原生能力做适配，少做那种看起来一致、实际不可靠的伪抽象。

### 6. 权限审批适合手机

Claude Code 需要工具审批时，TLive 会把权限请求变成飞书卡片。

手机上点“允许一次 / 本会话允许 / 拒绝”，比在手机 SSH 里盯着终端输入 `y/n` 舒服很多。

Codex 和 Pi 的权限模型不完全一样，TLive 会按它们能暴露出来的能力做适配。

### 7. 文件和图片也走 IM

TLive 支持飞书文件 / 图片进入会话，也支持 agent 通过 TLive MCP 工具把文件或图片发回飞书。

这对这些场景比较有用：

- 发截图让 agent 看 UI 问题。
- 发日志文件让 agent 分析。
- 让 agent 把生成的文件传回飞书。
- 在手机上快速查看结果。

## 和 SSH / 远程桌面的区别

SSH 和远程桌面很强，但用在手机上，很多时候还是在“把电脑搬到手机上”。

TLive 走的是另一条路：把 AI coding agent 接进 IM 工作流。

- 用飞书卡片做控制台。
- 用飞书话题做 session。
- 用群聊共享 agent。
- 用按钮处理权限。
- 用 server / client 管多台执行节点。
- 用文件和图片消息承载输入输出。

我希望手机端不用硬适配电脑终端，agent 可以直接进入 IM 的使用习惯里。

## 当前状态

项目还在快速迭代中，但主链路已经跑通：

- 飞书 / Lark 长连接
- 飞书工作台
- Claude Code / Codex / Pi 会话
- 远程执行节点
- 飞书话题 session
- 群聊 `@bot` 触发
- 话题内继续对话
- 最近会话恢复
- 权限审批
- 文件 / 图片
- client 自升级

离“无脑生产使用”还有距离，不过已经比较适合自己折腾、试用和二次开发。

如果你也想把 Claude Code / Codex / Pi 放进飞书，或者想在群里共享一个可协作的 AI 开发 agent，欢迎试试 TLive，也欢迎直接提 issue。

有什么 bug 都直截了当地提和修。
